# TransNetV2 场景检测迁移到 Web Worker

## 目标

将 `detectSceneTransNetV2` 的全部工作（视频解码 + Canvas 预处理 + ONNX WASM 推理 + 后处理）搬入 Web Worker，使主线程不再被阻塞。

## 方案选择：全量重建模式（方案 A）

Worker 内部从 `File` 对象开始重建完整的媒体处理链路：`BunnyMedia` → `BunnyClip` → 逐帧解码 → 预处理 → ONNX 推理 → 后处理。

所有依赖在 Worker 中均可用：
- `BlobSource(Blob)` — `Blob` 可通过 postMessage 传递
- `VideoDecoder` — WebCodecs API 在 Dedicated Worker 中可用
- `VideoFrame` — Worker 中可用
- `OffscreenCanvas` — 专为 Worker 设计
- `onnxruntime-web/wasm` — WASM 天然适合 Worker

## 涉及文件

### 新建文件

## 实现步骤

### Step 1: 创建 Worker 文件 `transnetv2.worker.ts`

Worker 负责：
1. 接收 `File` + 配置参数（threshold, minShotFrames, timeRange 等）
2. 内部 `new BunnyMedia(file)` → `new BunnyClip(bunnyMedia)` 重建解码链路
3. 逐帧 `tickN` → `toVideoFrame` → `preprocessFrame` → 累积到窗口 → `runner.run`
4. 通过 `self.postMessage` 回报进度
5. 通过 `self.postMessage` 返回最终 `bigint[]` 结果
6. 支持通过 message 取消检测

Worker 通信协议：

```
主线程 → Worker:
  { type: 'detect', file: File, timeRange: {...}, config: {...} }
  { type: 'abort' }

Worker → 主线程:
  { type: 'progress', event: TransNetV2ProgressEvent }
  { type: 'done', boundaries: string[] }     // bigint 序列化为 string
  { type: 'error', message: string }
```

注意：`bigint` 不能被结构化克隆，需要序列化为 `string` 传输（或使用 `Number`，因为这里的帧号不会超过 `Number.MAX_SAFE_INTEGER`）。

Worker 内部 import 列表：
```typescript
import { Input, BlobSource, ALL_FORMATS, VideoSampleSink } from 'mediabunny'
import { BunnyMedia } from '@/core/mediabunny/bunny-media'
import { BunnyClip } from '@/core/mediabunny/bunny-clip'
import { RENDERER_FPS } from '@/core/mediabunny/constant'
import { loadTransNetV2Runner } from './modelRunner'
import { TransNetV2Preprocessor, createTransNetV2Window } from './preprocessing'
import { probabilitiesToTimelineCutFrames } from './cutPostprocess'
import { ... } from './types'
```

关于 `modelManifest`：Worker 中需要 import `@/generated/model-manifest`。Vite Worker 使用 `type: 'module'` 时支持 `@/` alias。但更安全的做法是在消息中传入模型配置（path, version, chunks），避免 Worker 对 `@/generated` 的依赖。

实际上项目已有的模式是 `import ScriptExecutorWorker from './ScriptExecutor.worker.ts?worker'`，Vite 会正确处理 TS Worker 中的所有 import 包括 alias。

### Step 2: 改造 `scene-detector-transnetv2.ts`

将原来的 `detectSceneTransNetV2` 函数改造为 Worker 调度器：

1. 新建 `import TransNetV2Worker from './transnetv2/transnetv2.worker.ts?worker'`
2. 函数签名保持不变（`itemData` + `config`），但新增一个 `file: File` 参数
3. 内部创建 Worker、发送消息、监听结果
4. 将 `config.onProgress` 桥接到 Worker 的 progress 消息
5. 将 `config.signal`（AbortSignal）桥接到 Worker 的 abort 消息
6. 返回 `Promise<bigint[]>`

函数签名变更：
```typescript
// 之前
export async function detectSceneTransNetV2(
  itemData: UnifiedTimelineItemData,
  config: TransNetV2DetectorConfig = {},
): Promise<bigint[]>

// 之后
export async function detectSceneTransNetV2(
  itemData: UnifiedTimelineItemData,
  file: File,
  config: TransNetV2DetectorConfig = {},
): Promise<bigint[]>
```

### Step 3: 改造两个调用方

#### `useTimelineContextMenu.ts`（~line 528）
- 从 `timelineItem` 的 mediaItem 中获取 `File` 对象
- 通过 `bunnyMedia.getOriFile()` 获取原始文件
- 将 `file` 作为第二个参数传给 `detectSceneTransNetV2`

#### `VideoSceneSegmentsResolver.ts`（~line 48）
- 从 `mediaItem` 中获取 `File` 对象
- 同样通过 `bunnyMedia.getOriFile()`
- 将 `file` 作为第二个参数传给 `detectSceneTransNetV2`

### Step 4: Vite 配置确认

项目已有 `?worker` 导入模式（`ScriptExecutor.ts`）。Vite 默认支持 Worker 中使用 `@/` alias 和 TypeScript。无需额外配置。

如果遇到 `@/generated/model-manifest` 在 Worker 中无法解析的问题，备选方案是将模型配置（version, path, chunks）通过消息传入。

### Step 5: 浏览器兼容性处理

`VideoDecoder` 在 Worker 中的支持情况：
- Chrome 94+ ✅
- Safari 16.4+ ✅
- Firefox ❌（截至 2026 年仍未完全支持 WebCodecs）

降级策略：如果 Worker 中 `VideoDecoder` 不可用，回退到主线程的 `detectSceneAdv`（现有的快速检测）。这与当前代码中已有的回退逻辑一致。

## 风险点

1. **Worker 中的 `@/` alias**：需要 Vite 正确处理。项目已有 `ScriptExecutor.worker.ts` 先例可参考。
2. **ONNX WASM 路径**：`onnxruntime-web` 的 WASM 文件需要在 Worker 中正确加载。`ort.env.wasm.wasmPaths` 需要指向正确的 URL。
3. **`modelCache` 单例**：`loader.ts` 中的 `modelCache` 是模块级 Map，Worker 中会重新初始化，不会和主线程冲突。
4. **内存**：Worker 中 `BunnyMedia` + ONNX 模型会占用额外内存，但 Worker 结束后会释放。

## 不变点

- `TransNetV2DetectorConfig` 类型不变（除新增的内部字段）
- `TransNetV2ProgressEvent` 类型不变
- `probabilitiesToTimelineCutFrames` 逻辑不变
- 调用方的 `onProgress` 回调行为不变
- AbortSignal 取消机制不变（通过 message 桥接）
