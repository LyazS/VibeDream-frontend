# 智能分镜识别开发计划

## 目标

在 LightCut 前端实现一个基于 TransNetV2 的智能分镜识别能力，用 AI 镜头边界检测替换当前基于颜色直方图和峰值检测的方式。

第一阶段先把 TransNetV2 ONNX 模型放在前端 `public` 目录，快速打通纯前端识别和自动分割流程。第二阶段再把模型做成公开 npm 包分片，通过免费 CDN / npm 镜像多源下载、浏览器合并和缓存。

最终用户入口仍然是时间线视频片段右键菜单里的“智能分镜头检测”。用户选中当前时间线视频片段后触发检测，检测完成后直接调用现有分割命令拆分片段，不增加结果确认弹窗。

## 当前基础

当前项目已经有智能分镜入口：

- 入口在 `LightCut-frontend/src/core/composables/useTimelineContextMenu.ts`
- 当前默认调用 `detectSceneAdv`
- 检测结果会转成 `splitPoints`
- 最终通过 `unifiedStore.splitTimelineItemAtTimeWithHistory(clipId, splitPoints)` 分割
- 分割逻辑已经支持撤销/重做，由 `SplitTimelineItemCommand` 负责

当前已有三类前端检测算法：

- `scene-detector.ts`：HSV 直方图差异阈值
- `scene-detector-adv.ts`：Prominence 峰值检测
- `scene-detector-content.ts`：类似 ContentDetector 的像素差异检测

这些算法的共同问题是：它们主要看相邻帧颜色/亮度/边缘变化，对闪白、相机运动、主体快速移动、画面整体色调变化比较敏感，不是真正的镜头边界模型。

`/Users/airzostorm/Documents/test-findseg` 中已经有可参考的 TransNetV2 ONNX 实现：

- 模型文件：`backend/storage/models/transnetv2.onnx`
- 模型大小：约 30MB
- 模型 sha256：`c4d54a682bace32f25136ef83ca2c9d403e8f8193775efeb995172a0d95a8e0c`
- 输入：`input [1, 100, 27, 48, 3] tensor(float)`
- 输出：`[1, 100, 1] tensor(float)`，每帧一个 cut probability
- 默认阈值：`0.5`
- 当前 Python 逻辑按 100 帧窗口推理，不足 100 帧时重复最后一帧 padding

## 阶段一：public 模型版本

### 目标

先不处理模型公开分发复杂度，把 `transnetv2.onnx` 放进 `LightCut-frontend/public/models/transnetv2/`，在浏览器端用 `onnxruntime-web` 加载，完成当前时间线视频片段的智能分镜识别。

阶段一成功标准：

- 用户右键视频片段，点击“智能分镜头检测”
- 前端加载 public 目录里的 TransNetV2 ONNX 模型
- 当前选中片段被逐帧预处理为 48x27 RGB 输入
- 模型输出 cut probability
- 前端把 cut probability 转成切点帧号
- 直接调用现有 `splitTimelineItemAtTimeWithHistory`
- 失败时自动回退当前 `detectSceneAdv`

### 依赖

新增依赖：

```json
{
  "onnxruntime-web": "^1.x"
}
```

第一阶段建议优先使用 WASM execution provider。原因：

- TransNetV2 模型相对小，第一阶段重点是功能闭环。
- WebGPU 支持面和初始化失败场景更多。
- WASM 更适合先做稳定兜底。

后续如果性能不够，再尝试 `onnxruntime-web/webgpu` 和 `executionProviders: ['webgpu', 'wasm']`。

### 文件规划

新增：

LightCut-frontend/public/models/transnetv2/transnetv2.onnx

### 检测流程

入口仍然在 `detectSceneBoundaries()` 中。

阶段一将当前：

```ts
const boundaries = await detectSceneAdv(timelineItem, ...)
```

替换为：

```ts
const boundaries = await detectSceneTransNetV2(timelineItem, ...)
```

`detectSceneTransNetV2` 内部流程：

1. 从 `timelineItem.runtime.bunnyClip` 获取 `BunnyClip`
2. clone 一份 `BunnyClip`，避免影响主预览播放器
3. 根据 `timeRange.timelineStartTime` / `timelineEndTime` 只扫描当前片段范围
4. 逐帧调用 `tickN(currentFrameN, false, true)`
5. 把 `VideoSample` 转成 `VideoFrame`
6. 用 `OffscreenCanvas` 缩放到 `48x27`
7. 读取 RGBA 像素，转成 RGB float32，布局为 `[1, 100, 27, 48, 3]`
8. 每 100 帧调用一次 ONNX session
9. 不足 100 帧的最后窗口重复最后一帧补齐
10. 收集所有帧的 cut probability
11. 用 `threshold=0.5` 转成切点
12. 返回 `bigint[]`

返回值保持和现有检测函数一致：

```ts
Promise<bigint[]>
```

这样 `useTimelineContextMenu.ts` 中现有 split 逻辑不用重写。

### 后处理规则

第一阶段先复刻 `test-findseg` 的核心逻辑：

- `probability >= 0.5` 视为候选切点区域
- 连续高概率区域合并为一个切点，取区域中点
- 忽略第 0 帧和最后 1 帧附近切点
- 切点必须在当前片段范围内
- 返回的是时间线帧号，不是素材源文件帧号

最短镜头长度第一阶段使用固定值：

```ts
minShotFrames = 15
```

理由：

- Python 测试项目里 `TRANSNETV2_MIN_SHOT_FRAMES` 是 1，更适合后端素材索引。
- 前端时间线直接分割如果产生极短片段，用户体验会很差。
- 当前旧算法在菜单调用里也使用了 `minDistance: 15`，阶段一沿用这个体验。

### loading 和取消

复用现有 `unifiedStore.createLoading`。

阶段一 loading 文案建议：

- `加载智能分镜模型...`
- `分析视频帧 {current}/{total}`
- `AI 识别镜头边界...`
- `检测完成，正在分割 {count} 个场景...`
- `AI 模型不可用，已使用快速检测...`

取消逻辑继续使用 `AbortController`。

取消检查点：

- 模型加载前
- 每 30 帧预处理进度
- 每个 100 帧窗口推理前
- 推理后、分割前

取消时不执行分割，显示现有 `timeline.sceneDetection.cancelled`。

### 失败兜底

阶段一必须保留旧算法兜底。

兜底触发场景：

- `onnxruntime-web` 加载失败
- public 模型文件 404
- WASM 初始化失败
- 当前浏览器不支持必要 API
- 模型输入/输出 shape 不符合预期
- 推理过程中抛错

兜底行为：

1. loading 文案更新为“AI 模型不可用，已使用快速检测...”
2. 调用现有 `detectSceneAdv`
3. 旧算法成功则继续分割
4. 旧算法也失败才显示 error message

### 验收

阶段一验收条件：

- `npm run type-check` 通过
- `npm run build` 通过
- 对一个有明显剪切的视频片段，能检测出切点并自动分割
- 对一个无明显切点的视频片段，显示“未检测到明显的场景边界”
- 删除或改名 public 模型文件后，自动回退旧算法
- 检测过程中点击取消，不产生任何分割
- 分割后撤销/重做仍然正常

## 阶段二：公开 npm 分片模型版本

### 目标

将阶段一 public 单文件模型替换为免费公开分发方案：

```text
公开 npm 包
  ↓
多个 npm/unpkg 免费 CDN 镜像
  ↓
Web 端 fallback 下载分片
  ↓
浏览器合并 ArrayBuffer
  ↓
sha256 校验
  ↓
IndexedDB 缓存
  ↓
onnxruntime-web 创建 session
```

阶段二不改变用户操作流程，只替换模型加载来源。

### npm 包结构

建议包名：

```text
@lightcut/transnetv2-model
```

包结构：

```text
@lightcut/transnetv2-model
  package.json
  dist/
    manifest.json
    transnetv2-000.bin
    transnetv2-001.bin
    transnetv2-002.bin
    transnetv2-003.bin
    transnetv2-004.bin
```

分片大小建议：

```text
5MB - 8MB
```

不要用 30MB 或 50MB 单文件硬走免费 CDN。免费 CDN 容易遇到大小限制、同步不完整、CORS 不稳定或反滥用限制。

### manifest 格式

```json
{
  "name": "transnetv2",
  "version": "2026.05.13",
  "format": "onnx",
  "size": 31457280,
  "sha256": "c4d54a682bace32f25136ef83ca2c9d403e8f8193775efeb995172a0d95a8e0c",
  "input": {
    "name": "input",
    "dims": [1, 100, 27, 48, 3],
    "type": "float32",
    "layout": "NHWC"
  },
  "chunks": [
    {
      "name": "transnetv2-000.bin",
      "size": 8388608,
      "sha256": "..."
    }
  ]
}
```

注意：`layout` 用来说明模型输入含义，不参与 ONNX Runtime 的真实 shape 推断。

### CDN 源顺序

默认源顺序：

```ts
const cdnBases = [
  'https://npm.elemecdn.com/@lightcut/transnetv2-model@VERSION/dist/',
  'https://unpkg.zhimg.com/@lightcut/transnetv2-model@VERSION/dist/',
  'https://code.bdstatic.com/npm/@lightcut/transnetv2-model@VERSION/dist/',
  'https://hub.webcache.cn/npm/@lightcut/transnetv2-model@VERSION/dist/',
  'https://cdn.jsdelivr.net/npm/@lightcut/transnetv2-model@VERSION/dist/',
  'https://unpkg.com/@lightcut/transnetv2-model@VERSION/dist/',
]
```

运行时允许用环境变量覆盖：

```text
VITE_TRANSNETV2_MODEL_VERSION
VITE_TRANSNETV2_MODEL_CDN_BASES
VITE_TRANSNETV2_MODEL_MANIFEST_URL
```

### 下载策略

模型加载器逻辑：

1. 先检查 IndexedDB 是否已有当前 `version + sha256` 的完整模型
2. 命中缓存则直接返回 `ArrayBuffer`
3. 没命中则下载 `manifest.json`
4. 按 chunks 顺序下载分片
5. 每个分片优先从第一个 CDN 源下载
6. 单个分片下载失败时换下一个源
7. 分片下载完成后校验分片 sha256
8. 全部分片合并成一个 `ArrayBuffer`
9. 校验整包 sha256
10. 写入 IndexedDB
11. 返回给 ONNX Runtime

不要在阶段二直接依赖 HTTP Range。免费 CDN 对 Range、CORS exposed headers、缓存行为不一定稳定。分片文件是更可控的方式。

### 缓存策略

使用 IndexedDB，而不是只依赖 HTTP cache。

缓存 key：

```text
transnetv2:${version}:${sha256}
```

缓存 value：

```ts
{
  version: string
  sha256: string
  size: number
  createdAt: number
  model: ArrayBuffer
}
```

缓存清理：

- 只保留当前版本
- 新版本加载成功后删除旧版本
- 如果 sha256 校验失败，不写入缓存

### 发布脚本

新增脚本建议放在：

```text
scripts/prepare-transnetv2-model-package.mjs
```

职责：

- 读取 `test-findseg/backend/storage/models/transnetv2.onnx`
- 计算整包 sha256
- 按 5MB - 8MB 切片
- 计算每片 sha256
- 输出 npm 包目录
- 生成 `manifest.json`
- 生成最小 `package.json`

脚本输出目录可为：

```text
dist-model-packages/transnetv2-model/
```

这个目录不一定要提交到主仓库，具体是否提交由发布流程决定。

### 阶段二验收

- 删除 `public/models/transnetv2/transnetv2.onnx` 后，仍可通过 CDN 分片加载模型
- 首次加载能显示分片下载进度
- 第二次识别命中 IndexedDB，不再重复下载分片
- 手动让第一个 CDN 源失败时，能自动切到第二个源
- 任意分片 sha256 不匹配时，拒绝使用并尝试其他源或回退旧算法
- 模型整体 sha256 不匹配时，不创建 ONNX session
- 阶段一的分镜识别和自动分割验收项仍然全部通过

## 风险和处理

### 性能风险

纯前端逐帧预处理会消耗 CPU，长片段可能慢。

阶段一只处理当前选中片段，不做整素材批处理。进度每 30 帧更新一次，避免 UI 长时间无反馈。

### 内存风险

预处理如果一次性保存所有原始帧会占用大量内存。

实现时只保存 48x27 RGB 的缩小帧数据，或按 100 帧窗口流式处理。不要保存完整 `VideoFrame`，每帧处理后必须 `close()`。

### CDN 稳定性风险

免费 npm 镜像没有 SLA。

阶段二必须实现多源 fallback、sha256 校验和 IndexedDB 缓存。加载失败时回退旧算法，不能让功能完全不可用。

### 准确性风险

TransNetV2 主要识别硬切，可能不擅长所有转场或高度风格化剪辑。

第一版不做 UI 参数调节。后续如果用户反馈误切或漏切，再增加“灵敏度”设置，并把它映射到阈值和最小镜头长度。

## 后续可选增强

- 增加 WebGPU execution provider，设备支持时优先用 WebGPU
- 增加“灵敏度：保守 / 标准 / 激进”
- 增加检测结果预览，不立即分割
- 将分镜结果保存为片段 metadata，供 Agent 粗剪检索使用
- 给素材库增加“预分析分镜”能力，但这应是另一个阶段，不放在当前两阶段内
